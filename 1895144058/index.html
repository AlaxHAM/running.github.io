数据（Data）

​ 一句话概括 “描述事物的符号记录”

数据库（Database，DB）

​ 一句话概括 “按照数据结构来组织、存储和管理数据的仓库”

数据管理系统（Database Management System，DBMS）

​ 即为了对数据库进行管理而设计的软件系统，主要完成对数据库的操作与管理功能，实现数据库对象的创建、数据库存储数据的增删改查操作和用户管理以及权限管理等

类型

​ 根据基本的实现可以分为：关系型数据库，非关系性数据库（NoSQL）

关系型数据库

​ MySQL 就是一个非常流行的关系型数据库（Relational Database Management System），关系型数据的存储格式可以直观地反映实体间的关系，在关系型数据库中：数据以表格的形式出现，每一行是各个记录的名称，每一列为记录名所对应的数据域，多个行和列会形成一张表，多张表最后组成一个数据库

​ 关系型数据库除了基础的库，表，行，列之外，还有一些其他的术语：

​ 1）主键：在表中是唯一的，一张表只能拥有一个字段（列）设为主键

​ 2）外键：用于和其他的表形成关联，一张表中可以拥有多个外键，关联多个不同的表

​ 3）复合键：将多个列作为一个索引键

​ 4）索引：是对表中一列或多列的值进行排序的一种结构，用来快速查找指定的信息

​ 5）冗余：相同的数据存储了两次，冗余会降低性能

​ 6）参照完整性：及不允许一同不存在的实体，与实体完整性是关系模型必须满足的完整性约束条件， 目的是为了保证数据的一致性

​ 除了 mysql 一种关系型数据库外，当然还有 sqllite，db2，oracle，sqlserver 等，他们各自之间都存在不同的差异，但是对于大部分的 sql 语句来说在关系型数据库上都可以使用

非关系型数据库

​ 是随着技术的快速发展而不断扩展的一种数据库，简化了存储的数据结构并舍去了表结构，支持快速的查找需求，非关系型数据库可以分为四大类：键值对（key-value）存储结构（Redis），列存储，文档数据库存储（MongoDB），图形数据存储

SQL

​ SQL 是由 IBM 开发的，用户存储、查询、更新和管理数据库系统的语言，简单可分为 4 种：DDL，DQL，DML，DCL；除此之外还有 DPL（事务处理语言），CCL（指针控制语言）

DDL

​ 数据定义语言 ，即 数据库、表、视图、索引、存储过程，例如 CREATE ，DROP， ALTER

DQL

​ 数据查询语言 ，查询数据（SELECT）

DML

​ 数据操纵语言 ，即 插入数据（INSERT）、删除数据（DELETE）、更新数据（UPDATE）

DCL

​ 数据控制语言 ，即 控制用户的访问权限（GRANT）等

​ mysql 客户端：1）连接服务端 2）发送指令（sql 语句）

存储引擎

​ MySQL有很多不同的引擎，对于不同的应用场景可以选择合适的存储引擎来存储数据，例如：MyISAM，InnoDB，Memory，CSV 等等

​ MyISAM：用来管理非事务表，可以提供高速存储和检索，以及全文搜索能力，不支持外键，是 5.5 之前使用的默认存储引擎

​ InnoDB：支持事务，也支持外键，同时也支持崩溃修复能力和并发控制（MVCC），5.5 之后使用的默认存储引擎

​ Memory：即将数据全部会存在内存中，特点就是速度极快，但也因此不安全，而且表的大小受内存限制，也就是不能存储大量的数据

​ CSV：将数据使用逗号来隔开的形式存储在文件中

​ ‘.frm’ 文件存储的是 表的结构 ，在 lnnoDB 的存储中，‘.ibd’ 文件存储的表中的数据；MyISAM 中，表的数据文件后缀则是 ‘.MYD’，另外 MyISAM 还有一个索引文件 ‘.MYI’，如果是 Memory 作为使用的存储引擎则只会有 ‘.frm’ 的表结构文件，不会生成数据的文件

数据库的三大范式

​ 常用的是三大范式，除此之外关系型数据库有六个范式，巴斯-科德范式，第四范式，第五范式

​ 第一范式（1NF）：是对属性的原子性约束，要求字段（列）必须是不可分解的原子数据项；例如地址信息，如果是 “上海市，虹口区”，这样就不行，需要查出来，分为两个列 “省（市）” 和 “县（区）” 来进行存储

​ 第二范式（2NF）：满足第一范式的基础上，要求非主属性不能出现部分依赖主键，即每一列都与主键相关，不能只与主键的某一部分相关（主要针对联合主键而言）；例如订单表来说，一个订单会有多个产品，所以是 “订单号” 与 “产品号” 组成一个联合主键，但是对于 “金额” 和 “订单时间” 两个列，他们与产品并没有关系，如果这些存储在一张表中，就无法满足第二范式，所以可以拆出两个表将 “订单号” ，”金额” 和 “订单时间” 另组一张表

​ 第三范式（3NF）：满足第二范式的基础上，要求非主键字段不能出现依赖传递

子查询

​ 即将一个查询语句嵌套在另一个查询语句中，在在查询中可以包含：IN，NOT IN，ANY，ALL，EXISTS，NOT EXISTS 等关键字，同时也支持比较运算符如：=、!=、>、< 等

Code

​ 简单来说，利用子查询也涉及了多表或单个表需要进行多次过滤的操作，子查询的目的视为了给外部查询提供满足需要的结果集，来让外部函数做进一步的过滤等操作

Code

​ 子查询的分类：

​ 1）标量子查询：返回集中只包含单一值的标量

​ 2）列子查询：返回结果集是 N 行一列 的形式

​ 3）行子查询：返回结果集是 一行 N 列 的形式

​ 4）表子查询：即返回结果是 N 行 N 列

​ 针对 标量子查询，外部可以使用像 = ，< ，> 等之类的运算去比较值，如果子查询返回返回类型是 列，行，表的话，就不能使用比较运算符，而是使用 in，not in 之类的关键字

备份

​ mysql 的备份有：逻辑备份，物理备份，全量备份，以及增量备份

​ 逻辑备份，即备份表的结构，以及数据添加的 sql 语句，整个备份的过程会需要先对表做查询，再将结构与 sql 语句信息转为逻辑格式存储，适用于少量的数据备份和还原

​ 物理备份，即拷贝数据库的目录结构，包括数据文件，简单来说就像是对整个库做了一个镜像

​ 全量备份，即对某一时刻的所有数据做备份，相当于是给某一时刻数据库中的所有数据做了一个快照

​ 增量备份，即对某一个时间段中发生改变的数据做备份，增量备份的操作，需要开启 mysql 的二进制日志，用来对数据的改变做记录

​ mysql 的备份通常使用 mysqldump 来实现

存储过程

​ 把一段代码封装起来，当要执行的时候，通过调用存储过程来实现执行 sql（对于重复性的 sql 操作使用此方法可以提高效率 ，因为执行 sql 语句会存在一个编译过程 ，这种方法相当于将编译好的 sql 存在内存中，执行就直接从内存获取）

​ 优点：

​ 1）存储过程可封装，并隐藏复杂的商业逻辑

​ 2）存储过程可以回传值，且可以接受参数

​ 3）无法使用 select 指令

​ 4）可以用来数据校验

​ 缺点：

​ 1）存储过程，往往定制化于特定的数据库，当切换到其他的厂商的数据库上，就需要进行重写

​ 2）性能调校与撰写，受限于数据库系统

索引

​ 普通索引：可以在建表时就直接创建，也可以后续在需要的时候去添加（对于各种索引都可以在后续需要的时候去添加或删除），普通索引没有声明声明限制，纯粹是在单个字段上起到加速查询作用

Code

​ 唯一索引：即对使用唯一索引的字段附加了限制，要求该字段的值必须是唯一的，但是 允许字段的值为空 null，注意虽然允许值为 null，但是并不代表允许重复，因为 null 和 null 的比较永远是 False 的，所以并不会违反唯一的约束

Code

​ 主键索引：和唯一索引类似但是除了要求值 必须是唯一，并且不允许为空

Code

​ 联合索引：也可以称组合索引或是复合索引，即将多个字段列组合成一个索引，使用复合索引的好处就是避免了因为建立太多的单个索引，造成部分单个索引因为使用效率问题低而造成空间上的浪费

Code

​ 联合索引其实又可以分成，联合主键索引，联合唯一索引，即刚才的创建的联合普通索引，创建上都和原来得到单个字段列的方式一样，只是增加了需要的字段列

​ 最左前缀：使用联合索引的一个匹配的机制，例如组合索引为：(name,email) ，那么查询中使用的字段会去按照最左前缀的性质去匹配使用

​ 1）如果查询使用到 name and email，会使用该组合索引加速查询

​ 2）如果只使用 name 字段，也可以使用该组合索引

​ 3）如果只使用到 email 字段，无法使用组合索引

​ 所以，在创建复合索引的时候，一定要将频繁需要查询的字段放在复合索引的第一个

​ 覆盖索引：即在语句执行中，能从索引文件中直接获取数据（注意：这里的索引文件不是指磁盘上存储的索引文件，而是在内存中的，当语句中的字段是有索引的情况下，会优先在索引结构中去查找，然后根据结果回到表中去得到对应的数据，对于各个字段的索引都会有一个自己的树结构，这些结构会随着数据页加载到内存而被一起加载）

​ 所以，如果一个查询使用了索引的话，如果返回的数据是完全利用索引在内存中查询返回，而不是需要另外从磁盘中引入的话，那么使用的索引就可以称为是 覆盖索引，使用覆盖索引的好处，可以减少对磁盘的访问提升效率

​ 索引合并：即在语句中使用多个单例的索引进行搜索，最后在他们的查询后的结果进行合并，多个单例之前使用 and 或是 or 关联

​ 补充：查询时优化器的选择可能并不一定会是最优的，从而导致即使字段有索引，但最后查询的操作却很慢，所以可以通过强制使用索引来要求优化器去使用指定的索引，强制使用索引的语法是 force index()

sql

​ 索引的注意事项：

​ 1）避免直接使用 select *

​ 2）count(1) 或 count(字段列) 代替 count(*)

​ 3）创建表时，如果能使用 char 可以代替使用 varchar 的话，优先使用 char

​ 4）表的字段顺序应该是固定长度的字段优先

​ 5）利用组合索引代替使用多个单例索引

​ 6）尽量使用短索引

​ 7）使用连接的 join 去代替利用子查询的多表查询

​ 8）连接多表时，需要注意条件类型

​ 9）对于一些字段，例如性别这种，会有非常多一样的值的情况，不适合去建立索引

锁

​ mysql 的锁，大体使用方式上有两种，也就是常说的 乐观锁 和 悲观锁

​ 乐观锁：最常听到的解释是，乐观锁认为数据在操作的过程，一般不会造成冲突，也就说同一个数据的操作不会造成读取错误，但是在对数据更新时，会去判断数据修改时是否有其他事务对值做修改，如果产生冲突，则返回错误信息

​ 也就是说，乐观锁 是在应用层面上做的并发控制

​ 乐观锁的实现，常见的有两种：版本号控制，CAS 算法

​ 版本号控制：即在数据表中，添加一个 version 版本号的字段，每一次数据被修改后，version 的字段值就 +1，最后在提交这个更新前，会再去获取版本号，判断这次获取的版本和开始时获取的是否一致，一致就提交更新修改字段的值，不一致就表示在操作期间，数据被其他事务操作，这次的操作即使提交也不会进行数据更改，并返回信息给客户端

​ CAS 算法：compare and swap（比较和交换），也就说不通过锁来进行多线程的同步，做到非阻塞的同步

​ CAS 简单来说，涉及三个变量值，比如 V，A，B。V 表示主内存的值，A 表示各线程本地内存的值，B 表示各线程要修改后的值

​ 最后对于数据的更新，当且仅当 V = A 时，才会用 B 去更新 V，否则线程将不能进行数据更新，并且会不断 自旋（重试），之后自己的 V = A

​ 悲观锁：即与乐观锁相反，悲观锁认为每一次的数据操作都会出现冲突，为了保证线程安全，每一次的操作都需要进行加锁，这也直接导致数据操作会产生较多的耗时

​ 并且悲观锁是由数据库自己实现，也就是在数据库层面上做并发控制

​ mysql 中使用悲观锁的前提是，需要关闭自动提交 set autocommit = 0，默认是开启自动提交的，即事务提交后会自动将锁进行释放，让其他事务去获取锁

​ 对于 悲观锁，有两种实现方式：共享锁，排他锁

​ 共享锁：也叫 读锁，当一个数据出现并发操作，通过在 sql 语句后面加上 lock in share mode 来实现对一个资源的加锁操作，允许事务进行读取，但不能进行修改操作

​ 同时，如果一个事务给数据加上了 共享锁，其他的事务也只能对该数据加 共享锁，而不能加排它锁

​ 排他锁：也叫 写锁，与共享锁类似，但是语法上是在 sql 最后加上 for update 实现，当数据被加上 排他锁，其他事务同样不能对其进行操作，必须要先获取到被释放的 排它锁

​ 并且，一个数据被加上了 排他锁 后，其他事务不能对这个数据再去加其他的锁

​ 对于 mysql 来说，在 insert ，update，delete 的命令都是默认使用的 排它锁
